01

Power Platform & Dataverse foundations

Foundational

Everything you'll touch in Dynamics 365 sits on top of one shared foundation: the Power Platform. Get comfortable with these pieces first — you'll see all of them again in every later module.

The platform, at a glance

Five products share the same underlying data platform:

  • Power Apps — build canvas apps or model-driven apps (Dynamics 365 itself is a set of model-driven apps).
  • Power Automate — automate workflows with triggers and actions.
  • Power BI — build reports and dashboards for analysis.
  • Power Pages — publish external-facing websites backed by the same data.
  • Copilot Studio — build conversational AI agents.

What is Dataverse?

Dataverse is the secure, cloud-based data platform underneath all of it. Data lives in tables (rows and columns — you'll also hear the older terms entity, record, and field, which still show up throughout code and older documentation). Dataverse also carries the security model, business logic, and storage, so every app that reads from it behaves consistently.

Why teams rely on it instead of a bespoke database: it's easy to manage (metadata and data both live in the cloud), easy to secure (role-based access down to the row and column), rich in metadata (relationships are first-class, not an afterthought), and it comes with built-in logic and validation — calculated columns, business rules, and business process flows — so you rarely start from a blank slate.

Environments

An environment is an isolated space that holds its own apps, flows, and Dataverse database — typically one each for development, testing, and production. Treat environments as separate rooms: what you build in dev doesn't touch production until you deliberately move it there.

Solutions

A solution is a package of customizations — tables, forms, plugins, flows, and more — that you move between environments as a single unit. An unmanaged solution is editable and used while you build; a managed solution is locked and meant for production, so you import it rather than edit it directly.

Watch for this: "entity" and "table" mean the same thing, and so do "field"/"column" and "record"/"row". Microsoft renamed these in 2020, but code, error messages, and a lot of blog posts still use the old names.

Core CRM entities & relationships

Core concept

Almost every CRM conversation comes back to a handful of core tables. Learn these well: you'll query, customize, and troubleshoot them constantly.

  • Account — a company or organization you deal with.
  • Contact — an individual person, often linked to an Account.
  • Lead — an unqualified prospect, not yet vetted as a real deal.
  • Opportunity — a qualified, in-progress deal.
  • Campaign — a marketing effort aimed at a target audience.
  • Case — a customer service issue that needs resolving.
  • Activity — a logged interaction: email, call, task, or appointment.
  • Product — an item or service you sell.
Lookup fields: most of the connections below are built from lookup fields — a column on one table that points at a row on another. Once you can spot a lookup on a form, you can trace how any two tables relate.

Working with forms & views

Core concept

A form shows one record; a view shows a list of them. You'll spend most of your day moving between the two.

Form types

Form typeCustom componentsUse it for
Main formFull — PCF, JavaScript, business rulesThe full record: viewing, editing, everyday work
Quick CreateLimited — basic fields & business rulesFast entry from a side panel without leaving your place
Quick ViewLimited, read-onlyShowing fields from a related record inline
Card formNot customizableMobile or space-limited layouts

What getFormType() tells you

ValueModeMeaning
0UndefinedForm type not set
1CreateYou're creating a new record
2UpdateYou're editing an existing record
3Read-OnlyViewing only
4DisabledThe form is disabled
6Bulk EditEditing several records at once
Gotcha: if you create a record from inside a parent (say, a new Invoice under a Project Contract), Dynamics 365 sometimes pre-saves it to set up the relationship. The form still looks brand new to you, but getFormType() can already report 2 (Update) because the row technically exists. Check the URL for an ID if the form type surprises you.

Views

A public view is shared with everyone who has access to the table; a personal view is yours alone, for your own filters and columns. A subgrid is a form control that shows a filtered list of related records without leaving the current page — for example, an Account's open Opportunities shown right on the Account form.

05

Business rules & business process flows

Core concept

Before you ever reach for JavaScript, check whether a no-code option already covers what you need. These two are your first stop.

Business rules

A business rule validates data, shows or hides a column, sets a requirement level, or sets a value — without writing code. Scope it to the whole table and it runs both on the form and on the server, so it applies no matter how the data comes in.

Example: require a "Credit Limit Approver" only when Credit Limit is set above $1,000,000. Below that, the field stays optional.

Business process flows (BPFs)

A business process flow guides a user through a defined set of stages, shown as a bar across the top of the form. Each stage groups the columns someone should fill in before clicking Next Stage. The classic example is Lead to Opportunity to Quote to Order to Invoice.

  • By default, only System Administrator and System Customizer roles can edit a BPF's design — and each user only sees the BPFs their security role grants them.
  • A single BPF can span up to five different tables, and a table can have up to ten active BPFs, filtered by security role.
  • Conditions inside a BPF can branch up to five levels deep.

Locking a stage against manual changes

Sometimes a stage should only move forward through automation (a scoring model, a plugin, a flow) and never by a user dragging it. You can intercept the change client-side:

JavaScript · form web resource
var LeadSdk = window.LeadSdk || {};
(function () {
    this.formOnLoad = function (executionContext) {
        var formContext = executionContext.getFormContext();
        formContext.data.process.addOnPreStageChange(LeadSdk.formEvents.handleStageMovement);
    };

    LeadSdk.formEvents = {
        handleStageMovement: function (executionContext) {
            var formContext = executionContext.getFormContext();
            var bpfArgs = executionContext.getEventArgs();
            bpfArgs.preventDefault();

            var message = 'This stage is controlled automatically and cannot be changed manually.';
            formContext.ui.setFormNotification(message, 'WARNING', 'lockedBpf');
        }
    };
}).call(LeadSdk);

Customizing D365: scripts vs. plugins

Customization

Once business rules and BPFs aren't enough, you have two code-based options — and they solve very different problems.

AspectScripts (JavaScript)Plugins (C# / .NET)
Runs onClient-side, in the browserServer-side, on Dataverse
Triggers onForm events (load, save, change)Database events (create, update, delete)
Applies toThat form, in that browser sessionAll data entry: UI, imports, integrations
Best forField validation, dynamic show/hide, UI polishBusiness logic that must always apply, automation, integrations

A useful rule of thumb: reach for a script when you're improving the experience of filling out a form; reach for a plugin when the rule must hold true no matter how the record was changed.

Display rule vs. enable rule

These same two flavors show up again when customizing ribbon buttons:

AspectDisplay ruleEnable rule
EvaluatedOnce, when the form or grid loadsRe-evaluated as fields change on the form
ControlsWhether the button shows up at allWhether a visible button is clickable
Needs a refresh to update?YesNo — updates live
07

Plugin development deep dive

Developer topic

A plugin is a custom event handler: a compiled class in a plugin assembly (a .dll) that runs when a specific table and message combination is processed by Dataverse.

How a plugin gets called

The Execute method on your plugin class receives an IServiceProvider, which hands you the context, tracing service, and organization service.

Where in the pipeline it runs

Key interfaces you'll use

  • IServiceProvider — the entry point; you pull everything else out of it.
  • IOrganizationServiceFactory / IOrganizationService — create/retrieve/update/delete records from your plugin.
  • IPluginExecutionContext — carries InputParameters, OutputParameters, MessageName, PrimaryEntityName, Stage, Mode, Depth, UserId, entity images, and more.
  • ITracingService — your main debugging tool. Trace messages land in the Plug-in Trace Log, since you can't attach a live debugger to most sandboxed plugins.

Target, pre-image, and post-image

TermWhat it isTypical use
TargetThe data that triggered the plugin (an Entity for Create/Update, an EntityReference for Delete)Reading the new values being set
Pre-ImageA snapshot of the row before the operationComparing old vs. new values on an update
Post-ImageA snapshot of the row after the operationUsing the final values in a follow-up action

Registering a step

When you register a plugin, you're telling Dataverse exactly when to call it:

FieldWhat it controls
MessageThe operation that fires the plugin — Create, Update, Delete, Associate, or a custom action
Primary / Secondary entityWhich table (and, for two-table messages like Associate, the second table)
Filtering attributesOn Update, only fire when one of these specific columns changes — keeps unnecessary runs down
Execution orderControls run order when several plugins share the same stage, table, and message
DeploymentServer (the default, always applies) or Offline (legacy Outlook client)
Unsecure configurationPlain-text string passed to the plugin — visible to anyone with access to the step
Secure configurationEncrypted string, visible only to the user who registered it or an admin — use this for secrets

A simple, complete example

A post-operation plugin on Account's Delete message that removes related custom records:

C# · plugin class
public void Execute(IServiceProvider serviceProvider)
{
    var context = (IPluginExecutionContext)serviceProvider.GetService(typeof(IPluginExecutionContext));
    var serviceFactory = (IOrganizationServiceFactory)serviceProvider.GetService(typeof(IOrganizationServiceFactory));
    var service = serviceFactory.CreateOrganizationService(context.UserId);

    if (context.InputParameters.Contains("Target") && context.InputParameters["Target"] is EntityReference targetEntity)
    {
        if (targetEntity.LogicalName != "account")
            return;

        var query = new QueryExpression("new_relatedrecord")
        {
            ColumnSet = new ColumnSet(false),
            Criteria = new FilterExpression
            {
                Conditions =
                {
                    new ConditionExpression("new_accountid", ConditionOperator.Equal, targetEntity.Id)
                }
            }
        };

        var relatedRecords = service.RetrieveMultiple(query);

        foreach (var record in relatedRecords.Entities)
        {
            service.Delete(record.LogicalName, record.Id);
        }
    }
}

Walk through it slowly the first time: grab the context and service, confirm the message actually gave you a Target of the entity you expect, build a query for anything referencing it, then delete each match. That pattern — confirm, query, act — covers a large share of real plugins.

08

Ribbon customization

Customization

The ribbon (or command bar) is the row of buttons above a form or grid. You rarely hand-edit its XML directly — instead, you use the Ribbon Workbench, a community tool that runs inside XrmToolBox.

Building a display or enable rule

  1. Find the button's ID (the free Level Up browser extension reads this straight off the page).
  2. Open Ribbon Workbench, locate the button, and choose Customise Command.
  3. Add a new rule to the command, and pick a type — for example, ValueRule.
  4. Set its properties:
FieldMeaning
DefaultIf true, the rule is a fallback that applies unless another condition overrides it
InverseResultIf true, flips the evaluated result (true becomes false, and vice versa)
FieldThe logical name of the column being evaluated (e.g. statuscode)
ValueThe value the field is compared against

Wiring a script to a button

To run your own JavaScript from a ribbon button: add a Custom JavaScript Action to the command, point Library at your web resource, set Function Name to your namespaced function, and add a CrmParameter of type PrimaryControl — that parameter hands your function the form context directly.

Tool: XrmToolBox · Tool: Ribbon Workbench

09

Power Automate essentials

Automation

Power Automate builds automated workflows out of triggers (what starts a flow — a schedule, a new row, an incoming email) and actions (what the flow then does — update a record, send a notification, call an API).

The main flow types

Automated cloud flow

Runs whenever its trigger event happens — no one has to press anything.

Instant cloud flow

Runs when someone manually presses a button, from an app or the Power Automate mobile app.

Scheduled cloud flow

Runs on a recurring schedule — daily, weekly, whatever you set.

Desktop flow

Records and replays clicks and keystrokes on a desktop (RPA) — for systems with no API.

A simple example

You want a resource's status to flip from Available to Allocated automatically the moment someone fills in its allocation date. The flow: trigger on row modified for that table, filtered to when the allocation date column changes; then update the status column on that same row. No user has to remember to flip the switch by hand.

Reporting

Each of these four boxes is a "tile" — a single pinned visual on a dashboard.

Two more terms worth knowing: a semantic model is the underlying data, relationships, and calculations a report is built on; a KPI is a measurable value tracked against a target, like a sales goal of $1.2M against $1M actual.

Dynamics 365 Project Operations

Application deep dive

Project Operations connects sales, resourcing, project management, and finance into a single application — built for organizations that sell projects rather than off-the-shelf products. Think of it as everything you've learned so far, aimed at running a project from prospect to payment.

Heads up: Project Operations can be deployed a couple of different ways — running entirely on Dataverse, or connected to Dynamics 365 Finance for deeper ERP capability. Ask your mentor which setup your organization uses; the concepts below apply either way.

The four capability areas

Project sales

Price lists, project quotes, and profitability estimates — built on the same Lead/Opportunity concepts from Module 3.

Project planning

Task, board, and timeline views (via Project for the Web) to schedule and track deliverables.

Resourcing

Find, book, and manage the right people using the Schedule Board and Bookable Resources.

Time, expense & financials

Submit and approve time/expense entries, then bill using Time & Materials, Fixed Price, or Retainer contracts.

From opportunity to invoice

Resourcing, connected to what you already know

Resourcing reuses the Bookable Resource model: a system User can have a matching Bookable Resource record so they can be scheduled, and a Bookable Resource Category Assn links that resource to one or more Bookable Resource Categories (skills like "Consultant" or "Project Manager") so the Schedule Board can match the right person to the right task.

Everyday tools for CRM team members

Tooling

A handful of tools will show up in almost every task you do. Get them installed and get comfortable poking around in them.

  • XrmToolBox — a free desktop app hosting dozens of community-built tools for Dataverse, including the two below.
  • Ribbon Workbench — visually edit the ribbon/command bar (Module 8).
  • Plugin Registration Tool — register plugin assemblies and steps (Module 7).
  • Level Up — a browser extension for quick CRM debugging shortcuts, including grabbing a ribbon button's ID.
  • Advanced Find — the built-in query builder for searching and filtering records across the system.

Finding plugin trace logs

When a plugin misbehaves, your first stop is its trace output: go to Settings → Administration → System Settings → Customization and confirm Enable logging to plug-in trace log is set to All or Exception. Logs then appear under Settings → Plug-in Trace Log.

Two handy browser-console tricks

Both run straight in your browser's developer console while a Dynamics 365 form is open — useful for quick checks without writing a full web resource.

Get the name of the form you're looking at
var formName = Xrm.Page.ui.formSelector.getCurrentItem().getLabel();
console.log(formName);
Add a temporary filter to a lookup (e.g. show only active contacts)
(function () {
    var formContext = Xrm.Page;
    var control = formContext.getControl("primarycontactid");
    control.addPreSearch(function () {
        var filter = "<filter><condition attribute='statecode' operator='eq' value='0' /></filter>";
        control.addCustomFilter(filter);
    });
})();
