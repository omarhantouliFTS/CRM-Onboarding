01

Custom actions & APIs

APIs

Sometimes a piece of business logic needs to be callable from more than one place — a form script, a flow, and an external system all need to run the same "approve this quote" operation. Rather than duplicate that logic three times, you expose it once as a message that anything can call.

Bound actions vs. unbound actions

A bound action or function is attached to a specific table (and often a specific record), the way a method is attached to an object. An unbound one stands alone, not tied to any table — closer to a static utility function.

AspectBoundUnbound
Tied to a table?Yes — operates on a specific entity or entity collectionNo
Called via Web API as/api/data/v9.2/accounts(id)/Microsoft.Dynamics.CRM.new_ApproveAccount/api/data/v9.2/new_SendDailyDigest
Typical useAn operation naturally scoped to one record, e.g. "escalate this case"A general-purpose operation with no natural record owner, e.g. "recalculate exchange rates"

Custom action vs. custom API vs. plugin

All three run server-side logic, but they solve different problems. A plugin reacts to something that's already happening (a Create, an Update); a custom action or custom API is something you deliberately invoke on demand.

OptionTriggered byBest for
Plugin (event handler)A platform event: Create, Update, Delete, Associate, etc.Enforcing a rule that must always apply, no matter how the record changed
Custom actionExplicit invocation from a workflow, a button, JavaScript, or a flowA reusable, no-code-composable operation, quick to build via the classic UI designer
Custom APIExplicit invocation, same as an action, but backed by your own plugin classAnything needing real code logic, strong typing of parameters, and better performance than an action's XAML-driven workflow

In modern development, Custom API has mostly replaced custom actions for new work — it's faster (no workflow interpreter overhead), fully code-defined, and still callable from everywhere an action was. Custom actions still show up often in older solutions.

Custom API: overview & creation process

A custom API is a message definition (its name, whether it's bound, its parameters) paired with a plugin class that implements the actual logic.

  1. In the maker portal, go to your solution and choose New → Automation → Custom API.
  2. Name it with your publisher prefix (e.g. new_ApproveQuote), choose bound/unbound, and pick the bound entity if applicable.
  3. Add Custom API Request Parameters for each input, and Custom API Response Properties for each output — each gets a name and a type (String, EntityReference, Boolean, and so on).
  4. Write a plugin class implementing IPlugin that reads the request parameters off IPluginExecutionContext.InputParameters and writes results to OutputParameters.
  5. Register the plugin step against your Custom API's message name, same as any other plugin step.

Input & output parameters

FieldPurpose
Unique nameThe message name used to call it — must include your publisher prefix
Is functionFunctions are read-only and callable via GET (safe to cache); actions can write data and use POST
Is privateIf true, only callable internally (e.g. from a workflow/plugin), not exposed to external Web API callers
Allowed custom processing step typeWhether other plugins can hook into this API's own pipeline (None / Async only / Sync and async)
Request/Response parametersStrongly typed inputs and outputs, each with a logical name and a type

Calling it from every direction

The whole point of a custom action or custom API is that the same operation is reachable from anywhere:

OrganizationResponse response = service.Execute(request);

Power Automate deep dive

Automation

You've seen the basics of flows before; this module is about building ones that survive contact with real data — concurrent runs, transient failures, and flows that call other flows.

The four flow shapes

Automated

Fires on a Dataverse (or other connector) trigger — a row created, updated, or deleted. No human involved.

Instant

Fires when a user presses a button, from a model-driven app command bar, a canvas app, or the mobile app.

Scheduled

Fires on a recurrence (hourly, nightly, weekly) with no external trigger at all — good for batch cleanup or digest emails.

Business process flow trigger

Fires when a record moves into a specific BPF stage — useful for kicking off automation exactly when a process reaches a milestone.

Dataverse triggers and actions

The Dataverse connector's trigger, "When a row is added, modified or deleted," can be scoped tightly: pick the table, the change type, a scope (Organization, Business unit, Parent: business unit, User), and, critically, a filtering column list so the flow only fires when a column you actually care about changes — not on every unrelated save.

The matching actions — Add a new row, Update a row, Get a row by ID, List rows (with OData filter/sort/select) — are your CRUD toolkit inside the flow designer, without writing a single HTTP call by hand.

Scope, concurrency, and loops

A Scope action groups a set of steps so you can attach one "configure run after" behavior to the whole group, instead of wiring error handling onto every single action.

By default, an Apply to each loop runs its iterations in parallel (up to 20 concurrently), which is fast but means iteration 3 might finish before iteration 1. Turning on Concurrency Control lets you either force strictly sequential processing (degree of parallelism = 1) or raise the parallel limit up to 50.

SettingWhen to use it
Concurrency = 1 (sequential)Iterations must run in order, or they write to a shared resource that can't handle parallel writes (e.g. incrementing a running total on one record)
Concurrency raised above 20Independent iterations, high volume, and the downstream system can handle the burst (watch Dataverse API limits)
Default (20, parallel)Most loops over independent records

Error handling and retries

Every action has a built-in retry policy (default: 4 retries, exponential backoff, for a subset of transient errors) configurable per-action under the "..." menu → Settings. For your own error handling beyond the built-in retry, use "Configure run after" on a downstream action to make it run when the previous action failed, timed out, or was skipped — not just on success. That's how you build a "do X, and if it fails, log/notify/compensate" branch without a separate condition block.

"Configure run after" branches the flow based on the outcome of the previous step, not just a data condition.

Calling custom APIs and HTTP actions

To call a custom API or action from a flow, use the Dataverse connector's "Perform an unbound/bound action" (shown in Module 1), or for external systems, the generic HTTP action (premium) with a manually built request, or HTTP with Azure AD when the target needs an authenticated Azure AD token.

Child flows and environment variables

A child flow is a flow triggered only by another flow (trigger type: "When a Flow calls this Flow", with typed inputs/outputs) — the flow equivalent of extracting a shared function so five parent flows don't duplicate the same ten steps. Environment variables hold configuration that differs by environment (a SharePoint site URL, an approver's email, a feature flag) as a solution component with a default value plus a per-environment override, so the same flow definition deploys unchanged from dev to production while pointing at different endpoints.

Gotcha: connection references (not raw connections) are what make a flow portable between environments. If a flow was built against a raw, hardcoded connection instead of a connection reference, it will fail silently after import until someone manually reconnects it — see Module 8's deployment failures table.

03

Advanced customization

Customization

Once you're past business rules and simple scripts, four tools cover almost everything else you'll reach for on the front end.

Custom controls / PCF components

The Power Apps Component Framework (PCF) lets you write a fully custom control — a rating widget, a map picker, a signature pad — in TypeScript, and drop it onto a field or a dataset (a grid/subgrid) the same way you'd add any standard control. Unlike a web resource bolted onto a form with JavaScript, a PCF control is a first-class, packaged solution component with its own lifecycle methods (init, updateView, getOutputs, destroy) and typed manifest.

XML · PCF ControlManifest.Input.xml (excerpt)
<control namespace="Contoso" constructor="RatingControl" version="1.0.0" display-name-key="Rating Control">
  <property name="value" display-name-key="Value" of-type="Whole.None" usage="bound" required="true" />
  <resources>
    <code path="index.ts" order="1" />
  </resources>
</control>

Reach for a PCF control when a standard field or subgrid control can't express what the UI needs; reach for plain client scripting when you just need to react to events on standard controls.

Client scripts: best practices & formContext

Modern form scripts are written against the executionContext passed into every event handler, from which you get formContext — never the old global Xrm.Page, which breaks on multi-form scenarios like a Quick Create panel open alongside a main form.

JavaScript · formContext pattern
function onLoad(executionContext) {
    var formContext = executionContext.getFormContext();
    var statusControl = formContext.getControl("statuscode");
    var creditLimit = formContext.getAttribute("creditlimit").getValue();

    formContext.getAttribute("creditlimit").addOnChange(function (ctx) {
        var fc = ctx.getFormContext();
        fc.getControl("creditapprover").setVisible(fc.getAttribute("creditlimit").getValue() > 1000000);
    });
}
  • Namespace everything (e.g. Contoso.Account.onLoad) — never register a bare global function name, to avoid collisions across web resources.
  • Never hardcode a GUID, option set value, or environment URL — pull them from environment variables or a config table.
  • Always null-check a lookup or optionset attribute before reading .getValue() — an empty field returns null, not an empty object.
  • Keep heavy logic (queries, complex validation) server-side in a plugin; use client scripts for what the user needs to see change live.

Command bar customization beyond Ribbon Workbench

The modern command bar designer (built into the maker portal) now handles a large share of what used to require Ribbon Workbench: adding/reordering/hiding buttons, and writing show/hide logic with Power Fx expressions directly in the designer instead of hand-built Display/Enable Rule XML.

Power Fx · command bar visibility rule
Or(
    Self.Selected.Item.statuscode = 1,
    LegacyCommandBarContext.SelectedRecordsIds.Count = 0
)

Ribbon Workbench is still the right tool for anything the modern designer doesn't yet expose — deep JavaScript-driven enable rules, custom parameters, or editing the classic web client ribbon on older solutions.

Custom pages in model-driven apps

A custom page is a canvas-app-authored page (built with the same Power Fx and drag-and-drop designer as a canvas app) embedded inside a model-driven app — used for a bespoke dashboard, a wizard-style multi-step form, or a layout the model-driven form designer simply can't produce. It's added to the app's sitemap or opened from a command, and it can read/write Dataverse data the same as any canvas app, but lives and deploys inside the model-driven app's solution.

Choosing between them: business rule → no-code field logic. Client script → form-level behavior standard controls can't do. PCF → a genuinely new control type. Custom page → a layout the form/view designer fundamentally can't produce. Reach for the simplest one that solves the problem; each step up costs more to build and maintain.

Security model in practice

Security

Dataverse security answers one question for every single record: can this user do this action to this row? The answer comes from layering four concepts on top of each other: where the user sits in the org (business unit), what privileges they hold (security role), who owns the record, and any one-off shares.

Business units, security roles, and teams

Every user and team belongs to exactly one business unit; security roles grant privileges, business units scope how far those privileges reach.

A business unit (BU) is an organizational container used purely for security and reporting scope — not the same as a department in your HR system, though it's often modeled to match one. A security role is a named bundle of privileges (Create, Read, Write, Delete, Append, Assign, Share, per table) each with an access level. A team is a group of users that can itself own records and hold security roles, so a whole group gains access at once instead of assigning the role to each person individually.

User-owned vs. team-owned records

Most business tables are user- or team-owned, meaning every row has an Owner field (a user or a team) and access decisions key off that owner. A smaller set of tables (like most configuration/reference tables) are organization-owned, meaning there's no per-row owner and access is all-or-nothing across the org. Assigning a record to a team — instead of one person — is the standard pattern for shared queues (e.g. an unassigned Leads team) so any member can pick it up.

The five access levels

LevelGrants access to
NoneNothing — the privilege doesn't apply at all for this role
UserOnly records the user themselves owns
Business UnitRecords owned by any user or team in the user's own business unit
Parent: Child Business UnitsRecords owned within the user's business unit and any of its child business units
OrganizationEvery record in the environment, regardless of owner or business unit

Each privilege (Read, Write, Delete, and so on) gets its own access level within a role — it's entirely normal for a role to have Organization read but only User delete.

Sharing and cascading behavior

Sharing grants access to a specific record for a specific user or team, layered on top of whatever their role already allows — useful for a one-off exception without changing anyone's role. Shares can be revoked individually and don't affect any other record. Separately, relationship behavior (Referential, Referential: Restrict Delete, Parental, Custom) controls whether actions like assign, share, and delete cascade down to related child records — a Parental relationship (like Opportunity → Opportunity Product) cascades sharing and deletes automatically; a plain Referential one does not.

Column-level security

Some fields need to be locked down tighter than the record they live on — a Social Security Number, a negotiated discount, a salary. Field security profiles handle exactly this: enable Field Security on the column itself, create a Field Security Profile, set that field's Read/Update/Create permission on the profile, and add the users or teams who should have that elevated access. Everyone outside the profile sees the field as blank or masked, even if their table-level security role would otherwise let them read the record.

Debugging tip: when a user says "I can't see a record I should be able to see," check in this order: (1) do they have the right privilege at all on their security role, (2) is the access level high enough given the record's owner and business unit, (3) is there a share that should exist but doesn't, (4) is a field security profile hiding specific columns even though the record itself is visible.

Data integration & migration

Data

Almost every CRM implementation eventually needs to talk to something outside Dataverse — a legacy system's one-time data load, a nightly sync with an ERP, or a real-time lookup from a portal. Everything in this module is about the same handful of tools serving different points on that spectrum.

Dataverse Web API basics

The Web API is Dataverse's RESTful, OData v4 endpoint — the same API the CRM web client itself calls under the hood, which means anything you can do in the UI is reachable programmatically.

HTTP · retrieve a filtered, sorted list of accounts
GET https://yourorg.crm.dynamics.com/api/data/v9.2/accounts
    ?$select=name,revenue,telephone1
    &$filter=statecode eq 0 and revenue gt 1000000
    &$orderby=revenue desc
    &$top=25
Authorization: Bearer <access_token>
OData-MaxVersion: 4.0
OData-Version: 4.0
Accept: application/json

Standard verbs map the way you'd expect: GET to retrieve, POST to create, PATCH to update, DELETE to delete. Authentication is OAuth 2.0 against Microsoft Entra ID (Azure AD) — you'll register an app registration and use client credentials for a service-to-service integration, or an interactive/on-behalf-of flow when a real user is involved.

FetchXML vs. OData

AspectFetchXMLOData query
FormatXML query language, Dataverse-specificURL query parameters ($filter, $select, $orderby), REST-standard
Where it's usedViews, reports, RetrieveMultiple in plugins/code, XrmToolBox toolsWeb API calls, Power Automate's built-in filter/select fields
Aggregation & groupingFull support (sum, count, group by) directly in the queryLimited without $apply (OData aggregation extension, more verbose)
Readability for simple filtersMore verboseMore concise
FetchXML · same query as the Web API example above
<fetch top="25">
  <entity name="account">
    <attribute name="name" />
    <attribute name="revenue" />
    <attribute name="telephone1" />
    <order attribute="revenue" descending="true" />
    <filter type="and">
      <condition attribute="statecode" operator="eq" value="0" />
      <condition attribute="revenue" operator="gt" value="1000000" />
    </filter>
  </entity>
</fetch>

Importing data: dataflows, import wizard, SSIS/KingswaySoft

ToolBest forNotes
Import Data WizardA quick, one-time CSV/Excel load done by a maker, no dev involvedBuilt into the maker portal; maps columns visually; limited transformation logic
Power Query dataflowsRecurring loads from many source types with real transformation stepsSame Power Query engine as Excel/Power BI; can write straight into Dataverse tables
SSIS + KingswaySoftLarge-scale, complex, repeatable ETL — the de facto standard for big legacy migrationsThird-party connector for SQL Server Integration Services; handles huge volumes, upserts, and complex entity relationships that the wizard can't
Configuration Migration ToolMoving reference/config data (not transactional data) between environmentsFree Microsoft tool, schema-driven, good for seeding a new environment with lookup values
Migration tip: always load parent records before children (Accounts before Contacts before Opportunities) and disable unnecessary plugins/workflows during a bulk load, then re-enable and run any downstream automation deliberately afterward — otherwise you can trigger thousands of unwanted emails, flows, or plugin executions during the import itself.

Integration patterns: synchronous vs. asynchronous

Choose synchronous when the caller genuinely needs the result before it can continue (validating a payment before confirming an order). Choose asynchronous (a queue, a scheduled flow, an async plugin step) when the caller doesn't need to wait, when the downstream system is slow or unreliable, or when you want to protect the user's save operation from an external system's outage. Async plugin steps and Power Automate's own retry policy (Module 2) are the two most common asynchronous building blocks inside Dataverse itself.

06

Monitoring, debugging & performance

Operations

Shipping a change is only half the job — you also need to know when it breaks, and why, before a user tells you. This module is your troubleshooting toolkit.

Plugin trace log and system jobs

Every plugin execution can leave a trail. Turn it on under Settings → Administration → System Settings → Customization → Enable logging to plug-in trace log (set to All while actively debugging, Exception in steady-state production to limit noise), then read the results under Settings → Plug-in Trace Log, or with a tool like XrmToolBox's Plugin Trace Viewer for easier filtering. Each ITracingService.Trace(...) call you write in a plugin lands here, alongside the exception message and stack trace if it failed.

System jobs (Settings → System Jobs) show every asynchronous operation — async plugin steps, workflows, bulk operations — with a status (Waiting, In Progress, Succeeded, Failed) and, for failures, the error message. This is your first stop when something "should have happened but didn't" and no user-facing error appeared, since async failures don't interrupt whoever triggered them.

Power Automate run history

Each flow keeps a run history showing every trigger fire, every action's inputs/outputs, and exactly where a run failed. A failed run can be resubmitted directly from its history (useful after fixing a downstream outage) without waiting for the trigger condition to happen again naturally. For flows that fail intermittently, check the trigger conditions and concurrency settings first — the most common "it worked when I tested it, then failed at 2am" causes are a race condition between parallel runs or a transient API throttling error that wasn't retried.

Platform diagnostics and solution health

The Power Platform admin center exposes environment-level health: API request counts against your license entitlement, storage consumption by table, and (for Dataverse) a built-in Solution Checker that statically analyzes your plugins and workflows for known anti-patterns (blocking synchronous calls, missing null checks, unbounded queries) before you even ship. Run Solution Checker as a normal part of finishing a solution, the same way you'd run a linter before a pull request.

Common performance traps

TrapWhy it hurtsFix
A view or subgrid with no filters on a huge tableForces a large, unindexed scan every time the view loadsAdd filters, limit columns, and use indexed columns in sort/filter
Synchronous plugins doing external HTTP callsThe user's save blocks on network latency they can't see or controlMove the call to an async plugin step or a Power Automate flow instead
Plugins without filtering attributesFires on every column change, not just the ones that matter, multiplying executionsSet filtering attributes on the step registration via the Plugin Registration Tool
Heavy business process flows with many stages/branches on every form loadEach active BPF adds evaluation overhead on load and saveKeep BPFs to the stages users actually need; retire ones nobody uses
Retrieving full column sets ($select omitted) on large queriesPulls far more data over the wire than neededAlways specify only the columns you use, in FetchXML or $select
Nested loops calling the Web API per iterationN+1 query pattern; hundreds of round trips instead of one batchUse $batch requests or FetchXML with joins to fetch related data in one call
Debugging habit: when something is "slow" rather than "broken," reproduce it while watching the Plugin Trace Log and the flow run history side by side with a stopwatch. Nine times out of ten, the slow part is a single synchronous external call or an unfiltered query, not "Dynamics being slow" in general.

Application lifecycle management (ALM)

Operations

Everything in this guide — APIs, automation, customization, security, integrations, and monitoring — eventually has to move through a repeatable, low-drama release process. That process is ALM, and this module builds the architecture around it; Module 8 then covers the concrete solution mechanics that run inside it.

Environments and Dataverse architecture

Zoom out from a single dev/test/prod chain and a mature setup usually looks like this: one environment per active developer or workstream (or a small shared dev pool), a build/integration environment where everyone's work merges before testing, a UAT/test environment for stakeholders, and production. Each environment has its own Dataverse database, its own storage capacity, and its own set of connections — nothing about data or connections carries over automatically; only solutions (and the config data you deliberately migrate) do.

Pipelines in Power Platform

Power Platform Pipelines is Microsoft's built-in, low-code deployment tool: a maker with access clicks Deploy in their dev environment, picks a target stage, and the pipeline exports, packages, and imports the managed solution for them — with an approval gate if one's configured — without needing PowerShell, Azure DevOps, or admin-level access to the target environment. For teams needing more control (automated testing, code review gates, scheduled releases), the same export/import mechanics are driven by Azure DevOps or GitHub Actions pipelines using the Power Platform Build Tools / pac CLI instead.

Shell · pac CLI, the command-line equivalent of a pipeline stage
pac auth create --url https://yourorg-test.crm.dynamics.com
pac solution export --name ContosoCRM --managed true --path ./out/ContosoCRM_managed.zip

pac auth create --url https://yourorg-prod.crm.dynamics.com
pac solution import --path ./out/ContosoCRM_managed.zip --async true

Solution segmentation strategy

Don't ship one giant solution containing everything the org has ever built. Segment by ownership boundary and release cadence: a core solution for shared foundation components (tables, security roles) that changes rarely, and separate solutions per team or feature area (Sales customizations, Field Service customizations, Integrations) that each ship on their own schedule. Reference the core solution from the others rather than duplicating components into each.

StrategyProsCons
One monolithic solutionSimple to reason about early onEvery release touches everything; one team's change blocks another's; slow imports
Segmented by feature/teamIndependent release cadence; smaller, faster, lower-risk deploymentsRequires discipline around shared/core components and dependency management
Segmented by layer (data model vs. UI vs. logic)Clean separation of concernsMost changes touch more than one layer, so you're often deploying several solutions together anyway

Versioning and release notes

Every solution carries a version number (Major.Minor.Build.Revision, e.g. 3.4.12.0) that you bump on every export — treat it like semantic versioning: major for breaking changes, minor for new functionality, build/revision for patches and fixes. Pair every release with a short changelog (what changed, which tickets it closes, any manual post-import steps like reconnecting a flow) so that six months from now, someone can look at a production incident and know exactly what shipped and when.

Bringing it together: a healthy release, end to end, touches nearly every module in this guide: expose new logic through a custom API (Module 1), wire it into a flow with proper error handling (Module 2), keep the UI changes as simple as the problem allows (Module 3), lock it down with the right security roles (Module 4), validate any data migration steps (Module 5), watch the trace logs and run history after go-live (Module 6) — all shipped through a versioned pipeline (this module) built on top of the unmanaged-to-managed solution mechanics in Module 8.

08

Solution lifecycle & deployment

Deployment

A solution is the unit of deployment in Dynamics 365 — a package of tables, forms, views, plugins, flows, web resources, and security roles that you move between environments as a single, versioned bundle. Getting comfortable with the lifecycle of a solution is arguably the single most important operational skill on a CRM team, because it's the difference between a change landing cleanly and a Friday-afternoon production incident. This module is the hands-on companion to Module 7's bigger-picture ALM architecture.

Managed vs. unmanaged solutions

Every solution is either unmanaged (editable, used while actively building) or managed (compiled and locked, used to deploy). You build in an unmanaged solution in a dev environment, then export it as managed and import that managed copy into test and production. You never hand-edit a managed solution in place.

ConsiderationUnmanagedManaged
Editable after import?Yes, freelyNo — locked, components are read-only
Where it livesDevelopment environmentsTest, UAT, and production environments
Uninstall behaviorComponents stay behind (they merge into Default Solution)Clean uninstall — removes everything the solution added
LayeringN/A, it's the base layerCan be layered; a second managed solution can further customize (with managed properties allowing it)
Typical useActive development, prototyping, sandbox tinkeringEvery environment downstream of dev
Rule of thumb: production, and any other environment people share, should only ever receive managed solutions. If someone customizes production directly instead (unmanaged), there's no clean way to undo it later — those changes just merge into the environment's own Default Solution with no record of who added them or why. That silent drift — production quietly picking up changes dev never had — is the single most common cause of "it worked in dev but broke in prod" tickets.

Export/import between environments

The basic flow is always the same shape, whether you do it by hand in the maker portal or automate it: export the unmanaged solution from dev, convert/export it as managed, then import the managed version into the next environment up.

Two settings matter every time you import: Stage for upgrade lets you push the file to the server without applying it yet (useful for pre-staging a large import), and the choice between letting Dataverse process asynchronously (recommended for large solutions, tracked as a system job) versus waiting synchronously in the UI.

Solution patches and upgrades

Once a managed solution is in production, you have two ways to ship a change to it: a patch (a small child solution holding only new or changed components, good for a quick fix) or a full upgrade (a new version of the whole solution that replaces the old one on import, cleanly removing anything you deleted).

AspectPatchUpgrade
ContainsOnly new/changed components since the parent versionEvery component, at its current state
Can remove components?No — patches can only add or changeYes — anything missing from the new version is removed
Best forA fast, low-risk fix between planned releasesA planned release, or cleaning up patch sprawl
GotchaPatches pile up over time and get hard to trackMust be imported as "Stage for upgrade" then applied, or components are held in parallel until you finish the upgrade

Dependency checking & common deployment failures

Before Dataverse allows a delete, publish, or solution removal, it runs a dependency check — it won't let you delete a column that a form, view, business rule, or plugin step still references. That's a feature, not a bug, but it's also where deployments most often stall.

FailureUsual causeFix
Missing dependency on importSolution references a component (a table, a connection reference, a flow) that doesn't exist yet in the target environmentImport the missing solution/component first, or add it to this solution
Managed solution won't let you delete a fieldAnother managed layer, form, or plugin step still depends on itRun "Show Dependencies" first, remove the dependents, then delete
Import succeeds but a flow is turned offConnection references weren't remapped to the target environment's connectionsReconfigure connection references during/after import, then turn the flow back on
Plugin step errors after importAssembly version mismatch, or a secure/unsecure config value that only worked in the source environmentRe-register or update the step's configuration per environment
"Solution is missing required components"You added a component to your solution but forgot a component it depends on (e.g. an option set used by a new column)Use "Add Required Components" when adding anything to the solution
Watch for this: importing a solution does not automatically publish customizations in every case, and it does not automatically re-enable flows whose connections weren't remapped. Always check the import job's status and click through any flows/connection references it flags before calling a deployment done.

ALM basics: dev → test → prod

At minimum, plan for three environments: development (where unmanaged work happens, one per active workstream or developer), test/UAT (managed solutions only, where testers and stakeholders validate), and production (managed solutions only, real users and data). Nothing gets built directly in production. This is the ground-level version of the environment architecture covered in Module 7.
